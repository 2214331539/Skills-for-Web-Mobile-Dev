# 一、模块定位

## 1.1 模块名称

**用户认证模块 / Authentication Module**

## 1.2 模块目标

为 Web 系统提供统一、可复用、安全的用户身份认证能力，包括：

1. 用户注册
2. 用户登录
3. 用户退出
4. 登录状态保持
5. 密码找回
6. 密码修改
7. 邮箱/手机号验证
8. 图形验证码/短信验证码/邮箱验证码
9. 第三方登录
10. 多端登录管理
11. 登录失败限制
12. Token 管理
13. 用户安全审计
14. 账号冻结/禁用/注销

------

# 二、登录注册模块应具备的完整功能点

## 2.1 注册功能

注册是用户账号创建入口，应支持多种注册方式。

### 基础注册方式

1. 用户名 + 密码注册
2. 邮箱 + 密码注册
3. 手机号 + 密码注册
4. 邮箱验证码注册
5. 手机验证码注册
6. 第三方账号注册，例如微信、GitHub、Google

### 注册字段建议

| 字段            | 是否必填  | 说明                       |
| --------------- | --------- | -------------------------- |
| username        | 可选      | 用户名，允许后续修改       |
| email           | 可选      | 邮箱注册时必填             |
| phone           | 可选      | 手机号注册时必填           |
| password        | 必填/可选 | 验证码登录场景可不设密码   |
| confirmPassword | 必填      | 前端校验两次密码是否一致   |
| captcha         | 条件必填  | 图形验证码                 |
| emailCode       | 条件必填  | 邮箱验证码                 |
| smsCode         | 条件必填  | 短信验证码                 |
| inviteCode      | 可选      | 邀请码注册                 |
| agreeTerms      | 必填      | 是否同意用户协议和隐私政策 |

### 注册时需要校验

1. 用户名是否重复
2. 邮箱是否重复
3. 手机号是否重复
4. 密码强度是否达标
5. 验证码是否正确
6. 验证码是否过期
7. 是否频繁注册
8. IP 是否异常
9. 设备指纹是否异常
10. 是否同意用户协议

### 注册成功后的处理

1. 创建用户基础信息
2. 创建用户认证信息
3. 记录注册日志
4. 自动登录，或跳转登录页
5. 发送欢迎邮件/站内通知
6. 初始化用户默认配置
7. 初始化默认角色，例如普通用户
8. 生成 Access Token 和 Refresh Token

------

## 2.2 登录功能

登录是认证模块的核心，应支持多种登录方式。

### 登录方式

1. 用户名 + 密码登录
2. 邮箱 + 密码登录
3. 手机号 + 密码登录
4. 手机验证码登录
5. 邮箱验证码登录
6. 第三方 OAuth 登录
7. 扫码登录，可选
8. 单点登录 SSO，可选

### 登录流程

```
用户输入账号密码
        ↓
前端基础校验
        ↓
后端校验账号是否存在
        ↓
校验账号状态
        ↓
校验密码 / 验证码
        ↓
检查登录失败次数
        ↓
生成 Token / Session
        ↓
记录登录日志
        ↓
返回用户信息和登录凭证
```

### 登录时需要判断的账号状态

| 状态     | 说明                  |
| -------- | --------------------- |
| active   | 正常可用              |
| pending  | 未完成邮箱/手机号验证 |
| locked   | 临时锁定              |
| disabled | 管理员禁用            |
| deleted  | 用户已注销            |
| banned   | 风控封禁              |

### 登录失败处理

1. 密码错误次数 +1
2. 连续失败达到阈值后短暂锁定
3. 异常 IP 登录触发图形验证码
4. 多次失败后发送安全提醒
5. 不要明确提示“账号不存在”或“密码错误”，避免账号枚举攻击

推荐提示：

```
账号或密码错误，请重新输入。
```

而不是：

```
该邮箱未注册。
```

------

## 2.3 退出登录功能

退出登录不能只在前端删除 Token，还应通知后端使 Refresh Token 或 Session 失效。

### 退出方式

1. 当前设备退出
2. 所有设备退出
3. 指定设备退出
4. 后台强制用户下线

### 退出时处理

1. 删除前端本地登录态
2. 后端注销当前 Refresh Token
3. 记录退出日志
4. 可选：清除服务端 Session

------

## 2.4 找回密码功能

找回密码通常基于邮箱或手机号验证。

### 找回密码流程

```
输入邮箱/手机号
        ↓
发送验证码
        ↓
验证验证码
        ↓
设置新密码
        ↓
使旧 Token 失效
        ↓
重新登录
```

### 找回密码安全要求

1. 验证码有效期一般 5 到 10 分钟
2. 验证码只能使用一次
3. 限制验证码发送频率
4. 限制同一 IP 请求频率
5. 修改密码后让所有旧登录态失效
6. 发送安全提醒

------

## 2.5 修改密码功能

修改密码针对已登录用户。

### 修改密码字段

| 字段            | 说明       |
| --------------- | ---------- |
| oldPassword     | 原密码     |
| newPassword     | 新密码     |
| confirmPassword | 确认新密码 |

### 修改密码规则

1. 必须校验原密码
2. 新密码不能与旧密码相同
3. 新密码必须满足强度规则
4. 修改后可选择退出其他设备
5. 修改密码后记录安全日志

------

## 2.6 验证码功能

验证码可以分为三类：

1. 图形验证码
2. 邮箱验证码
3. 短信验证码

### 验证码适用场景

| 场景         | 推荐验证码      |
| ------------ | --------------- |
| 注册         | 邮箱/短信验证码 |
| 登录失败多次 | 图形验证码      |
| 找回密码     | 邮箱/短信验证码 |
| 修改敏感信息 | 邮箱/短信验证码 |
| 异地登录     | 邮箱/短信验证码 |
| 高频请求     | 图形验证码      |

### 验证码安全要求

1. 有效期限制
2. 发送频率限制
3. 验证失败次数限制
4. 使用后立即失效
5. 不在日志中明文记录验证码
6. 验证码建议只存哈希值

------

## 2.7 登录态管理

推荐采用：

```
Access Token + Refresh Token
```

### Access Token

用于访问业务接口。

特点：

1. 有效期短
2. 一般 15 分钟到 2 小时
3. 前端请求时携带
4. 被盗后风险较低

### Refresh Token

用于刷新 Access Token。

特点：

1. 有效期较长
2. 一般 7 天到 30 天
3. 应支持服务端主动失效
4. 每次刷新可以轮换 Refresh Token

### Token 存储建议

| 存储位置        | 优点                | 风险            |
| --------------- | ------------------- | --------------- |
| localStorage    | 使用简单            | 容易受 XSS 影响 |
| sessionStorage  | 浏览器关闭后失效    | 仍受 XSS 影响   |
| HttpOnly Cookie | JS 无法读取，更安全 | 需要处理 CSRF   |
| 内存变量        | 安全性较好          | 刷新页面后丢失  |

较推荐方案：

```
Access Token 放内存或短期 Cookie
Refresh Token 放 HttpOnly + Secure Cookie
```

------

## 2.8 第三方登录功能

第三方登录适合提升注册转化率。

### 支持平台

1. GitHub
2. Google
3. 微信
4. QQ

### 第三方登录流程

```
点击第三方登录
        ↓
跳转第三方授权页
        ↓
用户授权
        ↓
第三方回调 code
        ↓
后端用 code 换取 access_token
        ↓
获取第三方用户信息
        ↓
绑定或创建本地用户
        ↓
生成本站登录态
```

### 第三方登录注意事项

1. 一个用户可以绑定多个第三方账号
2. 第三方账号可以绑定已有邮箱用户
3. 第一次第三方登录时需要创建本地用户
4. 解绑第三方账号前要确保用户仍有其他登录方式
5. 不要把第三方 access_token 明文暴露给前端

------

## 2.9 多设备登录管理

用户可以查看和管理自己的登录设备。

### 功能点

1. 查看当前登录设备
2. 查看历史登录记录
3. 退出当前设备
4. 退出其他设备
5. 退出所有设备
6. 标记可信设备
7. 异常设备提醒

### 设备信息

| 字段         | 说明         |
| ------------ | ------------ |
| deviceName   | 设备名称     |
| browser      | 浏览器       |
| os           | 操作系统     |
| ip           | 登录 IP      |
| location     | 登录地区     |
| lastActiveAt | 最后活跃时间 |
| isCurrent    | 是否当前设备 |
| isTrusted    | 是否可信设备 |

------

## 2.10 账号安全中心

完整认证模块建议配套一个账号安全中心。

### 可展示内容

1. 邮箱是否绑定
2. 手机号是否绑定
3. 密码是否已设置
4. 第三方账号绑定情况
5. 最近登录记录
6. 当前在线设备
7. 是否开启二次验证
8. 账号风险等级

### 可操作内容

1. 修改密码
2. 绑定邮箱
3. 绑定手机号
4. 解绑第三方账号
5. 开启二次验证
6. 退出其他设备
7. 注销账号

## 3.2 核心用户用例

## 用例 1：用户注册

### 用例名称

用户注册账号

### 参与者

游客

### 前置条件

用户未登录

### 基本流程

1. 用户进入注册页面
2. 用户输入邮箱/手机号/用户名
3. 用户设置密码
4. 用户获取验证码
5. 用户输入验证码

| 异常情况     | 系统处理           |

| 验证码错误   | 提示验证码错误     |
| 验证码过期   | 提示重新获取       |
| 请求过于频繁 | 提示稍后再试       |

用户账号创建成功。

用户登录系统

游客

# 四、数据库表设计

下面是一个相对完整、可复用的数据库设计。你可以根据项目复杂度删减。

------

# 4.1 用户主表：users

用于保存用户的核心身份信息。

```
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',

    username VARCHAR(50) UNIQUE COMMENT '用户名',
    nickname VARCHAR(100) COMMENT '用户昵称',
    avatar_url VARCHAR(500) COMMENT '头像地址',

    email VARCHAR(255) UNIQUE COMMENT '邮箱',
    email_verified TINYINT DEFAULT 0 COMMENT '邮箱是否已验证',

    phone VARCHAR(30) UNIQUE COMMENT '手机号',
    phone_verified TINYINT DEFAULT 0 COMMENT '手机号是否已验证',

    status VARCHAR(30) NOT NULL DEFAULT 'active' COMMENT '账号状态：active/pending/locked/disabled/deleted/banned',

    register_source VARCHAR(50) COMMENT '注册来源：web/mobile/github/google/wechat',
    register_ip VARCHAR(64) COMMENT '注册IP',

    last_login_at DATETIME COMMENT '最后登录时间',
    last_login_ip VARCHAR(64) COMMENT '最后登录IP',

    password_updated_at DATETIME COMMENT '密码最后修改时间',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted_at DATETIME NULL COMMENT '软删除时间'
);
```

说明：

`users` 表只保存用户主体信息，不建议直接把密码字段放在这里。更好的做法是将密码、第三方登录、登录方式等放到独立表中，便于扩展。

------

# 4.2 用户密码表：user_credentials

用于保存密码哈希。

```
CREATE TABLE user_credentials (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '凭证ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',

    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希',
    password_salt VARCHAR(255) COMMENT '密码盐，使用bcrypt/argon2时可不单独存',

    password_algo VARCHAR(50) DEFAULT 'bcrypt' COMMENT '密码算法：bcrypt/argon2/pbkdf2',
    password_version INT DEFAULT 1 COMMENT '密码策略版本',

    failed_login_count INT DEFAULT 0 COMMENT '连续登录失败次数',
    locked_until DATETIME NULL COMMENT '锁定到期时间',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    UNIQUE KEY uk_user_id (user_id)
);
```

安全建议：

1. 不存明文密码。
2. 不存可逆加密密码。
3. 推荐使用 `bcrypt` 或 `argon2`。
4. 不建议自己设计加密算法。
5. 密码校验失败时增加失败次数。
6. 成功登录后清空失败次数。

------

# 4.3 第三方账号绑定表：user_oauth_accounts

```
CREATE TABLE user_oauth_accounts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT 'ID',
    user_id BIGINT NOT NULL COMMENT '本地用户ID',

    provider VARCHAR(50) NOT NULL COMMENT '第三方平台：github/google/wechat/qq/apple',
    provider_user_id VARCHAR(255) NOT NULL COMMENT '第三方平台用户ID',
    provider_username VARCHAR(255) COMMENT '第三方用户名',
    provider_avatar VARCHAR(500) COMMENT '第三方头像',

    access_token_encrypted TEXT COMMENT '加密后的第三方access_token，可选',
    refresh_token_encrypted TEXT COMMENT '加密后的第三方refresh_token，可选',
    token_expires_at DATETIME COMMENT '第三方token过期时间',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    UNIQUE KEY uk_provider_user (provider, provider_user_id),
    KEY idx_user_id (user_id)
);
```

注意：

如果业务不需要调用第三方 API，可以不保存第三方 access_token，只保存绑定关系。

------

# 4.4 验证码表：verification_codes

用于邮箱验证码、短信验证码、找回密码验证码等。

```
CREATE TABLE verification_codes (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '验证码ID',

    target VARCHAR(255) NOT NULL COMMENT '接收目标：邮箱或手机号',
    target_type VARCHAR(20) NOT NULL COMMENT 'email/phone',

    scene VARCHAR(50) NOT NULL COMMENT '场景：register/login/reset_password/bind_email/bind_phone',
    code_hash VARCHAR(255) NOT NULL COMMENT '验证码哈希',

    expires_at DATETIME NOT NULL COMMENT '过期时间',
    used_at DATETIME NULL COMMENT '使用时间',

    send_ip VARCHAR(64) COMMENT '发送请求IP',
    verify_attempts INT DEFAULT 0 COMMENT '验证尝试次数',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

    KEY idx_target_scene (target, scene),
    KEY idx_expires_at (expires_at)
);
```

不建议直接存验证码明文，建议存验证码哈希。

------

# 4.5 用户会话表：user_sessions

用于管理用户登录态、多设备登录、Refresh Token。

```
CREATE TABLE user_sessions (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '会话ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',

    refresh_token_hash VARCHAR(255) NOT NULL COMMENT 'Refresh Token哈希',
    access_token_jti VARCHAR(100) COMMENT 'Access Token唯一ID，可选',

    device_id VARCHAR(255) COMMENT '设备ID',
    device_name VARCHAR(255) COMMENT '设备名称',
    user_agent TEXT COMMENT '浏览器UA',
    os VARCHAR(100) COMMENT '操作系统',
    browser VARCHAR(100) COMMENT '浏览器',

    ip VARCHAR(64) COMMENT '登录IP',
    location VARCHAR(255) COMMENT '登录地区',
```

# 八、安全特性设计

这是登录注册模块最重要的部分。

------

## 8.1 密码安全

必须做到：

1. 密码不明文存储
2. 使用 bcrypt / argon2 哈希
3. 设置密码复杂度规则
4. 限制弱密码
5. 修改密码后旧 Token 失效
6. 不在日志中打印密码
7. 前端不要把密码保存在本地
8. 后端接口使用 HTTPS

密码规则建议：

```
长度 8-32 位
至少包含字母和数字
推荐包含特殊字符
不能与用户名、邮箱过于相似
不能使用常见弱密码
```

------

## 8.2 Token 安全

建议：

1. Access Token 短有效期
2. Refresh Token 长有效期
3. Refresh Token 存 HttpOnly Cookie
4. Refresh Token 服务端存哈希
5. 支持 Refresh Token 轮换
6. 支持主动失效
7. 支持踢下线
8. 使用 HTTPS 传输
9. JWT 中不要放敏感信息

JWT Payload 示例：

```
{
  "sub": "10001",
  "username": "test_user",
  "roles": ["user"],
  "iat": 1710000000,
  "exp": 1710007200,
  "jti": "token_unique_id"
}
```

不要放：

```
{
  "password": "...",
  "phone": "...",
  "idCard": "..."
}
```

------

## 8.3 防暴力破解

措施：

1. 单账号登录失败限制
2. 单 IP 登录失败限制
3. 登录失败后要求图形验证码
4. 连续失败后临时锁定账号
5. 增加请求间隔
6. 记录失败日志
7. 异常行为通知用户

推荐策略：

```
同一账号连续失败 5 次：要求验证码
同一账号连续失败 10 次：锁定 15 分钟
同一 IP 高频请求：限制访问
```

------

## 8.4 防账号枚举

账号枚举是指攻击者通过提示信息判断某个邮箱/手机号是否已注册。

不推荐：

```
该邮箱不存在
该手机号未注册
```

推荐：

```
账号或密码错误
如果该邮箱存在，我们会发送重置邮件
```

找回密码时也不要明确暴露账号是否存在。

------

## 8.5 防 CSRF

如果使用 Cookie 存 Token，需要考虑 CSRF。

防护方式：

1. SameSite Cookie
2. CSRF Token
3. 检查 Origin / Referer
4. 重要操作要求二次验证

Cookie 建议：

```
HttpOnly
Secure
SameSite=Lax 或 Strict
```

------

## 8.6 防 XSS

防护方式：

1. 前端避免直接渲染不可信 HTML
2. 对用户输入进行转义
3. 设置 Content-Security-Policy
4. Token 尽量不放 localStorage
5. 后端返回数据前进行必要过滤

------

## 8.7 防短信轰炸和邮箱轰炸

验证码发送必须限制频率。

建议：

```
同一手机号 60 秒内只能发送一次
同一手机号每天最多 10 次
同一 IP 每小时最多 30 次
同一设备每天最多 20 次
```

------

## 8.8 敏感操作二次验证

以下操作建议要求二次验证：

1. 修改密码
2. 修改邮箱
3. 修改手机号
4. 绑定第三方账号
5. 解绑第三方账号
6. 注销账号
7. 提现、支付等资金相关操作

------

## 8.9 审计日志

必须记录：

1. 登录成功
2. 登录失败
3. 修改密码
4. 找回密码
5. 绑定邮箱
6. 绑定手机号
7. 绑定第三方账号
8. 退出所有设备
9. 账号锁定
10. 管理员禁用账号
